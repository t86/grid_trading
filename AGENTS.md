# 编码准则

写代码、改代码、review 或重构时遵守这些约束。它们是行为准则，不是准确率承诺。

## 优先级

这些规则是默认执行门禁，不是建议。它们优先于“做完整一点”、未来扩展性、顺手重构、UI 顺手接入和抽象洁癖。

如果实现开始明显变大，先停下来说明更小的实现路径和取舍，再继续。

## 分支与部署基线

- 默认直接在 `main` 上开发、验证和准备部署，保持工作区干净。
- 除非用户明确要求开分支、建 worktree 或做实验功能，不要主动切分支或新建 worktree。
- 生产部署前确认本地 `main` 与 `origin/main` 的关系；需要同步时优先使用 `git pull --ff-only origin main`。
- 为了便于远端服务器直接更新，完成且验证通过的生产相关改动应落在 `main`，再按用户要求提交、推送或部署。

## 先想清楚再写

- 实现前明确说明关键假设；如果存在多个会影响实现的解释，不要静默选择。
- 需求不清且猜测有风险时，停下来问清楚。
- 如果用户要求会引入不必要复杂度，要直接指出更简单的路径和取舍。

## 简单优先

- 只写解决当前问题所需的最小代码。
- 不添加未被要求的功能、配置项、抽象层、兼容分支或“未来可能有用”的弹性。
- 如果实现明显可以从 200 行变成 50 行，先简化再继续。

## 外科手术式修改

- 只改和当前请求直接相关的文件与代码行。
- 匹配现有代码风格；不要顺手重排、重命名、格式化或重构无关代码。
- 清理本次改动制造出的未使用 import、变量、函数或注释；不要删除历史遗留死代码，除非用户明确要求。
- 发现无关技术债可以说明，但不要塞进同一个 patch。

## 目标驱动执行

- 非平凡任务先转成可验证目标，再动手。
- 多步骤任务用简短计划：`步骤 -> 验证：检查方式`。
- 修 bug 优先写出失败测试或最小复现，再修到通过。
- 重构要保持行为检查前后都通过。
- 持续循环到验证通过；如果卡住，明确说明阻塞点和剩余风险。

## 减少重复探测

- 开始前先读当前上下文、工作树状态和已知事实；能从现有输出判断的，不重复跑同类命令。
- 切分支、建 worktree、找 Python/测试工具前，先说明为什么当前环境不能直接用；没有明确收益时留在当前工作树。
- 生产相关改动先确认目标主机、运行路径和 commit，再决定是否需要本地切换；不要在本地旧分支和生产代码不一致时盲改。
- 命令失败后先归纳失败原因，再换一种最小验证方式；不要连续试相近命令。
- 如果发现自己在重复确认同一类信息，停下来汇总已知结论和下一条必要命令。

## Wangge 现货损耗口径

- 复核 Binance APP `交易分析` 里的现货交易对损耗时，默认按交易对自身口径计算：`损耗 = 买入总额 - 卖出总额 - 当前持仓数量 * 最新价`，`万U损耗 = 损耗 / (买入总额 + 卖出总额) * 10000`。
- 这个口径不回加或抵扣手续费；BNB 手续费、USDT 手续费、BNB 余额变化和合约对冲浮盈亏只能作为单独口径列出，不能拿来抵消 APP 交易分析损耗。
- 如果同时需要看账户总资产影响，必须把 `APP交易分析口径`、`手续费/BNB口径`、`合约对冲盈亏口径` 分开报告，不要混成一个损耗数。

## Wangge 合约与冻结仓位边界

- 普通底舱回归、库存偏离修正、补底仓或刷量提速时，不得用合约仓位做平衡；底舱回归只能通过现货挂单或停止等待完成。
- 只有明确属于冻结仓位账本的动作，才允许联动合约；例如 `spot_freeze` 生成的 `freeze_long_hedge` / `freeze_short_hedge`、`pending_contract_repair`、冻结释放或冻结配对释放。
- 判断冻结仓位是否合理时，按冻结账本核对：`expected_short = base_hedge_qty - frozen_long_qty + frozen_short_qty`。实际合约空仓应与该值在容差内一致。
- 底舱恢复优先于刷量；启用 `spot_base_restore_only` 时，只允许保留能把现货底舱拉回 `neutral_base_qty` 的订单，回到容差内后不得重新进入 normal 双边网格。
- `spot_freeze_long_total_cap_notional` 和 `spot_freeze_short_total_cap_notional` 表示分侧冻结累计总量硬上限，不是触发阈值；某侧已冻结名义值达到或超过该上限后，不得继续冻结该侧。
- 汇报时必须区分“冻结仓位导致的合约变化”和“底舱回归导致的合约变化”。前者可以接受，后者违反运行边界。
